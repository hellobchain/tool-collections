## 项目介绍

### 项目需求描述：智能合同起草助手（Agent）

#### 1. 项目背景与目标
**背景**：传统合同起草依赖法务人工翻阅模板、手动填充条款，效率低且易遗漏关键风险点。
**目标**：构建一个**智能合同起草Agent**，允许用户上传标准模板文档（.doc/.docx）并输入自然语言需求，Agent通过LLM理解意图，自动生成一份结构完整、条款合规、可直接使用的定制化合同草案。

#### 2. 核心用户角色
- **业务人员**：无法律背景，提出业务诉求（如“我需要一份软件外包开发合同，付款分三期”）。
- **法务/合规人员**：上传内部标准模板，审核并微调生成结果。

#### 3. 核心功能流程（端到端）
1.  **模板上传**：用户上传1个或多个合同模板文件（支持.doc/.docx）。
2.  **需求输入**：用户通过文本输入合同关键要素（如：合同类型、双方信息、标的物、金额、付款方式、特殊要求等）。
3.  **智能解析（Agent预处理）**：
    - 提取模板中的**结构骨架**（章节、条款、层级）。
    - 识别模板中的**可替换变量**（如“甲方”“总金额”“交货日期”）。
    - 识别模板中的**可选条款**（如“保密条款”“违约责任方式”）。
4.  **需求理解与映射**：LLM将用户自然语言需求转化为**结构化参数**（如：`{party_a: "XX科技", payment_terms: "30%-40%-30%", ...}`）。
5.  **智能起草生成（核心）**：
    - 基于模板骨架，将结构化参数填充至对应变量位置。
    - 根据用户需求，动态**增删/改写**条款（如需求提到“源代码归属”，则强化知识产权条款）。
    - 对缺失的必要条款（如争议解决）进行**智能补全**并给出提示。
    - 保持输出格式与原模板风格一致（字体、缩进、编号等）。
6.  **结果预览与导出**：生成完整合同草案，支持在线预览，并一键导出为标准.docx文件。

#### 4. 关键能力要求（Agent设计要点）
- **文档解析能力**：能准确读取.docx中的段落、表格、页眉页脚、样式，保留格式元数据。
- **模板抽象层**：设计一套中间数据结构（如JSON Schema）来描述模板的“可变部分”和“逻辑规则”，使LLM与模板解耦。
- **LLM提示工程**：设计多步Prompt Chain（如：需求结构化 → 条款匹配 → 冲突检测 → 生成润色），而非单次大Prompt。
- **合规性保障**：内置常见法律风险提示规则（如“违约金超过30%可能无效”），在生成时附加风险注释。
- **可追溯性**：生成的每一条款需标注来源（模板原文 / LLM新增 / LLM改写），便于人工复核。

#### 5. 输入输出规范
- **输入**：
  - 模板文件（.doc/.docx，最大10MB）
  - 需求文本（支持500字以内，含关键要素）
- **输出**：
  - 完整合同草案（.docx格式，保持原模板样式）
  - 附带一份《条款变更说明》（Markdown或JSON），列出：新增条款、删除条款、修改条款、填充变量值、风险提示。

#### 6. 非功能性需求
- **响应时间**：起草过程不超过60秒（含LLM推理）。
- **准确率**：关键变量填充准确率 ≥ 98%（如公司名、金额、日期格式）。
- **容错性**：当用户需求与模板严重冲突时，Agent需主动反问澄清，而非强行生成。
- **数据安全**：模板和合同内容不用于模型训练，支持私有化部署。

#### 7. 技术约束与扩展性
- **LLM**：建议使用支持长上下文（≥32K）的模型，以便处理复杂模板。
- **格式处理**：推荐使用 `python-docx` 或 `unstructured` 库解析，结合 `markdown` 作为中间编辑格式以降低LLM处理难度。
- **扩展点**：未来需支持多模板组合（如从多个模板中挑选条款合并），请在设计时预留Agent工具调用（Tool Calling）接口。

#### 8. 验收标准（示例）
- 上传一份“采购合同”模板，输入“采购100台服务器，总价500万，预付30%，验收后付70%”，生成草案中付款条款正确填充，并自动补全验收标准条款。
- 上传一份“租赁合同”模板，输入“租期3年，月租金2万，含免租期1个月”，生成草案中租金计算正确，且自动添加“提前解约”条款。
- 所有生成结果均无格式错乱，导出可正常打开。

---

#### 9. 输出格式

在 API 接口设计中，无论业务处理成功或失败，HTTP 状态码均统一返回 200。响应体结构采用统一格式，具体字段定义如下：

- **code**：业务状态码，类型为整型。当值为 `0` 时表示请求处理成功；其他非零值则表示失败，具体错误类型由不同错误码区分。
- **msg**：结果描述信息，类型为字符串。成功时通常返回 `"success"`，失败时返回对应的错误提示信息。
- **data**：业务数据负载，类型为对象。成功时返回具体的业务数据；失败时通常返回空对象或 `null`。

示例响应结构如下：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

这种设计将 HTTP 传输层与业务状态解耦，便于前端统一处理响应，同时通过 `code` 字段精细化区分业务异常。
