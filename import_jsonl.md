
## 需求：Go 后端新增 JSONL 流式读取接口

### 1. 目标
为现有 Go 后端服务新增一个 HTTP 接口，用于**流式读取**服务器本地的一个 JSONL 文件，逐行解析并返回数据，支持**大量数据**场景。

### 2. 背景
- 文件格式：**JSONL**（每行一个独立 JSON 对象）
- 文件位置：服务器本地路径（可配置）
- 数据量：**巨大**（单文件可能 GB 级），**不允许**一次性全量加载到内存
- 通用结构：每行 JSON 可反序列化为一个通用 Go `struct`（字段已知，见下方示例）

### 3. 功能要求

#### 3.1 HTTP 接口定义
- **Method**：`GET`
- **Path**：`/api/v1/data/stream`
- **请求参数**（可选）：
  - `offset`（int，默认 0）：从第几行开始读取（支持断点续读）
  - `limit`（int，默认 1000，最大 10000）：本次最多返回的行数
  - `file`（string，必填）：指定读取哪个 JSONL 文件（或使用配置中的默认文件）

#### 3.2 流式读取逻辑
- 使用 `bufio.Scanner` 或 `json.Decoder` 逐行读取
- 每读取一行，立即解析为通用结构体
- **边读边返回**（SSE 或 Chunked Transfer），而不是等全部读取完再返回
- 若客户端断开连接，应**立即中止**读取并释放资源

#### 3.3 通用结构体示例
```go
type Record struct {
    ID    string          `json:"id"`
    Time  int64           `json:"time"`
    Data  json.RawMessage `json:"data"`  // 或具体结构
}
```
> 若字段不确定，则使用 `map[string]interface{}`，但优先使用已知结构以提升性能。

#### 3.4 错误处理
- 文件不存在 → 返回 `404`
- 参数非法（如 limit 超限）→ 返回 `400`
- 读取过程中发生 I/O 错误 → 返回 `500` 并记录日志
- 解析失败（非 JSONL）→ 跳过该行并记录错误日志（可配置是否跳过或终止）

#### 3.5 性能要求
- 内存占用：**恒定**（与文件大小无关）
- 响应时间：首字节延迟 < 200ms（文件打开 + 首行读取）
- 吞吐：不低于 10MB/s 读取速率

### 4. 非功能性约束
- 兼容现有路由框架（Gin / Echo / 原生 net/http，以项目现有为准）
- 日志：使用项目现有 log 库，记录读取行数、耗时、错误
- 可观测性：增加 Prometheus 指标（可选，如读取总行数、错误计数）
- 安全性：仅允许读取**白名单目录**下的文件，防止路径遍历攻击

### 5. 交付物
- 新增接口的完整 Go 代码（含 handler、reader、参数校验）
- 单元测试（mock 文件系统，测试 offset/limit 及错误场景）
- 接口文档（Swagger/OpenAPI 注释）
- 本地启动后的 curl 验证示例

### 6. 验收标准
- [ ] 能正常读取 1GB+ JSONL 文件，内存占用 < 50MB
- [ ] 支持 offset/limit，返回行数准确
- [ ] 客户端断开后，服务端 goroutine 在 3s 内退出
- [ ] 所有错误场景有对应 HTTP 状态码和错误信息
- [ ] 单元测试覆盖率 > 80%
